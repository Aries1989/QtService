10 # Features {#qtservice_readme_label_0}
11 - Allows you to create system (and user) services for various systems
12  - Provides a single Interface to implement only once without having to care to much about the platform
13  - Actual platform specifc stuff is provided via plugins, making it easy to support other backends as well
14  - Most backends support native logging (i.e. journald, syslog, winevents, ...)
15  - Supports socket-activation (for systemd & launchd)
16  - Interfaces can be used to handle general service events
17 - Easy testing/debugging by simply switching the service backend via the start arguments
18 - Provides very basic service control to, e.g. get the status of a service or start/stop it
19 - Provides generic callbacks to allow realtively easy access to platform specific features
20 - Comes with a "Terminal frontend mode" which allows to create a CLI interface that communicates with the actual service without any extra coding needed
21  - Terminals will connect to the running service. If none is running, they can start it (on some platforms)
22  - Terminals send their command line arguments to the running service
23  - A connected terminal can send input to the master (via stdin) and receive output (stdout)
24  - Terminals can disconnect by beeing closed, leaving the service running (even the one that started the master)
25  - The Service gets a "Terminal" handle for each connected terminal, allowing parallel connections. Each terminal implements QIODevice, making communication easy
26  - For all this no extra code is needed from your side
27 - Supported service backends:
28  - Systemd (For most modern Linux Distros)
29  - Windows Services
30  - Launchd (Tested on MacOs, but should theoretically work on other unixes as well)
31  - Android Services
32  - "Standard Services" - A platform independed dummy backend that runs services as a normal proccess - useful for testing, debugging or as fallback.
33 - QtCreator template to easily create a service project, with configuration file templates for each platform
34 
35 **Note:** What this framework does explicitly *not* handle is "service installation" - as that is extremly different on all platforms and cannot be easily generalized. Performing the service installation, registration or whatever the platform needs is up to the developer. However, all the files that are needed as metadata for a service on each platform are provided via the QtCreator project template, which makes it significantly easier to deploy your service, as all thats left is to correctly "apply" those.
36 
37 Details on what exactly is supported by each backend can be found in the doxygen documentation under the "Related Pages" tab.

69 # Usage {#qtservice_readme_label_2}
70 The C++ part of the Service is relatively simple. All you have to do is implement the `QtService::Service` interface and start it in your main. The most important methods of the service are `start` and `stop` - which are called when the service was started or stopped...
71 
72 ```.cpp
73 class TestService : public QtService::Service
74 {
75  Q_OBJECT
76 
77 public:
78  explicit TestService(int &argc, char **argv) : //the reference here is important!
79  Service{argc, argv}
80  {}
81 
82 protected:
83  CommandMode onStart() override {
84  qDebug() << "Service was started";
85  return Synchronous; //service is now assumed started
86  }
87 
88  CommandMode onStop(int &exitCode) override {
89  qDebug() << "Stop received";
90  // do some complicated stopping stuff asynchronously that takes some time...
91  QTimer::singleShot(3000, this, [this](){
92  emit stopped(EXIT_SUCCESS);
93  });
94  return Asynchronous; // service is still stopping until "stopped" has been emitted
95  }
96 };
97 
98 int main(int argc, char *argv[]) {
99  // IMPORTANT: do NOT create a QCoreApplication here - this is done internally by the backends!
100  // also, do nothing else in the main besides setting the serices properties! Any setup etc. must all be
101  // done in the onStart method!!!
102  TestService service{argc, argv};
103  return service.exec();
104 }
105 ```
106 
107 **Rules of usage:**<br/>
108 You should always follow the following rules when using/creating a service. You can also find this
109 list in the QtService::Service class documentation:
110 
111 - Do not create a QCoreApplication yourself - this is done internally
112 - Use the same constructor signature to pass the main arguments. It is very important that the argc
113 argument is passed as reference! Passing as value will crash your application
114 - Do nothing else in the main besides setting the serices properties that need to be set early! Any
115 setup etc. must all be done in the Service::onStart method!!! (The properties that need to be set
116 early all have a hint in their documentation)
117 - Never call QCoreApplication::quit (or QCoreApplication::exit)! Use Service::quit instead
118 - Do not rely on QCoreApplication::aboutToQuit, as this may not be emitted at all, or on some
119 arbitrary point. Put all cleanup code in the Service::onStop method
120 - Be careful with Service::preStart - only use it when you have no other choice
121 
122 Actually running the service depends on your choosen backend. When testing with the standard backend, you can simply run the service without any parameters, or the backend explicitly specified:
123 
124 ```
125 /path/to/service --backend standard
126 ```
127 
128 To run with a system backend, you need to register the executable somehow in the service system of your operating system. The trick here is to register it with the backend as parameter. So for example, on windows you would register the service as `\path\to\service --backend windows`. For more details check the documentation or create a service project from the template - it will create service files that already do this for you.

130 ## Terminals {#qtservice_readme_label_3}
131 To allow terminals, you must set the `QtService::Service::terminalActive` property to `true` and implement the `QtService::Service::terminalConnected` and optionally the `QtService::Service::verifyCommand` methods. The service is started the same way as usual. Internally, you will get the terminals as soon as they have connected and can then process their arguments and communicate with them.
132 
133 To start a terminal client, simply prepend `--terminal` as well as the backend the service is running on to the command line. It is recommended to create a small wrapper script for each platform that takes care of these parameters. On linux, for example, you would create a shell script like below that simply starts the executable with the additional parameters:
134 
135 ```.sh
136 #!/bin/sh
137 # use as "service-cli <arguments>" (assuming this script is named service-cli)
138 exec /path/to/service -platform minimal --backend systemd --terminal "$@"
139 ```
140 
141 (The platform argument is recommended to not depend on any windowing system)

143 ## Service Control {#qtservice_readme_label_4}
144 The `QtService::ServiceControl` allows you to control services by sending commands to them and retrieving the status. However, what exactly is possible greatly varies for each platform. Always use `QtService::ServiceControl::supportFlags` to figure out what you can actually do on the current platform. You can also check the doxygen documentation to get an overview over all the backends and their features.
